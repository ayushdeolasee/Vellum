Overall verdict

Do not merge yet

The pull-model architecture is worth keeping, but it is not reliably cost-effective yet. Retrieval results are appended to the assistant message, persisted, and sent again in later prompts. Gemini thinking tool calls discard required signatures. OpenAI and ChatGPT can silently end incomplete when reasoning consumes the fixed output budget.

Fix findings 1–4 before merge. Findings 5–10 should follow in the same stabilization pass because they affect hidden spend, long-document trustworthiness, or provider parity. The remaining items are smaller but high-leverage.

Vetted findings

Findings, ordered by leverage

1. Keep retrieval output out of persisted conversation prose

Critical
IntroducedCost · CorrectnessEffort MFix risk MediumConfidence High

getPageText returns an uncapped full page. Every provider stores that result in actionResults; AiStore appends all results under “Actions:”, persists the assistant message, and the next turn embeds the last ten messages. One retrieval therefore becomes repeated paid prompt text, visibly leaks internal tool output, and can grow later turns beyond the old full-context design.

AiToolEngine.swift:153–163 — returns the full page string. AiStore.swift:538–549 — appends every action result and persists it. AiPrompts.swift:62–64 — resends the last ten persisted messages.
Fix direction
Represent tool traces separately from user-visible/persisted messages. Persist only the final answer and short mutation summaries. Cap retrieval outputs by token budget, and add a prompt-size regression test covering two follow-up turns after multiple reads.

2. Preserve Gemini thought signatures across tool turns

Critical
IntroducedProvider correctnessEffort MFix risk MediumConfidence High

The raw REST client extracts only functionCall from each returned part, reconstructs new model parts, and drops sibling thoughtSignature data. Google requires thought signatures to be returned with function-calling turns for thinking models. The next Gemini request can therefore fail after the first tool call.

GeminiClient.swift:73–86 — reads the parts but retains only the nested function call. GeminiClient.swift:94–114 — reconstructs model parts without signatures. Google Gemini thinking guide — REST callers must preserve whole signed parts; signatures are required for function calls.
Fix direction
Retain and replay the exact returned model parts, including signature fields, rather than rebuilding them from the function call dictionary. Test a fixture containing a signed function call followed by a function response.

3. Use the current Gemini 3 thinking contract

High
IntroducedCost · Provider correctnessEffort SFix risk LowConfidence High

The user-facing thinking picker maps every Gemini 3 model to numeric thinkingBudget. Current Gemini documentation says Gemini 3 should use thinkingLevel; numeric budgets are a Gemini 2.5 control and can produce unexpected Gemini 3 Pro behavior. “High” also selects 24,576 thought tokens, which are billed, without any cost signal in the UI.

GeminiClient.swift:40–53, 189–203 — emits numeric budgets for model ids containing “3”. AiSettingsPanel.swift:140–147 — exposes modes without provider-specific explanation. Google Gemini thinking guide — Gemini 3 uses minimal/low/medium/high levels; thought tokens are billed.
Fix direction
Map Gemini 3 to thinkingLevel with model-capability guards, keep thinkingBudget only for 2.5, and show a concise “higher cost and latency” hint for Medium/High.

4. Handle Responses API truncation instead of silently finalizing it

Critical
IntroducedCorrectness · CostEffort SFix risk LowConfidence High

OpenAI and ChatGPT set max_output_tokens to 2,048 while also allowing medium/high reasoning. That limit includes both invisible reasoning and visible text. Their SSE switches ignore response.incomplete, so a reasoning-heavy request may end with a partial answer—or no text—and be presented as a normal completion.

OpenAIClient.swift:35–55, 67–91 ChatGPTClient.swift:46–69, 75–99 OpenAI Responses streaming reference — the limit includes reasoning and visible output; incomplete responses are a distinct event.
Fix direction
Handle response.incomplete explicitly and surface its reason. Use an output budget appropriate to selected reasoning effort, or keep the visible-answer reserve independent in product policy. Add SSE fixtures for completed, failed, and incomplete streams.

5. Add AI pipeline tests before expanding provider behavior further

High
Introduced gapTest coverageEffort MFix risk LowConfidence High

The branch adds five network clients, OAuth, caching, two retrieval tools, streaming, cancellation, thinking modes, and Keychain migration. The only new tests cover selectable message rendering. No test asserts provider payload shape, tool-loop history, prompt growth, cancellation, or migration behavior.

Tests/SelectableMessageTests.swift:1–82 — renderer/layout tests only. project.pbxproj:552–558 — test target contains only PDF persistence and selectable-message suites.
Fix direction
Extract request-body construction and SSE decoding into pure, injectable units. Add Swift Testing suites for all provider payloads, signed Gemini turns, incomplete Responses, retrieval limits, prompt-size growth, cancellation, Keychain failures, and cache breakpoint counts. Keep live networking out of unit tests.

6. Cancel hidden requests and extraction when the document changes

High
IntroducedCost · ConcurrencyEffort SFix risk LowConfidence High

A tab/document change resets activity to idle but does not cancel the stored send task. The user can start another request while the old provider call continues invisibly. Whole-document extraction also has no cancellation check; Task.yield() does not cancel it.

ContentView.swift:65–71 — clears document context on identity change. AiStore.swift:295–321 — cancellation is used by Clear, not document reset. PdfSelectionBridge.swift:617–645 — whole-document loop never checks cancellation.
Fix direction
Cancel and nil the active task before resetting document state, cancel any previous task when registering a replacement, and use throwing cancellation checks inside extraction batches. Treat cancellation as lifecycle, not an error banner or tool result.

7. Provide a bounded path for trustworthy long-document summaries

High
IntroducedRetrieval qualityEffort LFix risk MediumConfidence High

The model sees one current page, can read at most 16 individual pages, and receives search snippets from at most eight pages. Common requests such as “summarize this 80-page report” or “compare every occurrence” cannot be completed faithfully. The prompt nevertheless says reads are free and to use as many as needed, so the assistant may answer confidently from a sample.

AiToolEngine.swift:21–32 — 16-read ceiling. AiToolEngine.swift:227–266 — search returns one hit per page, maximum eight pages. tool-mode-native.md:12–25 — promises broad document access without disclosing limits.
Fix direction
Add a budget-aware batch/range read or hierarchical map-reduce summary tool with explicit coverage metadata. The final answer should be able to say “covered 80/80 pages” or clearly disclose sampling. Do not simply raise the read count; that recreates the original cost problem.

8. Make page images explicit and deduplicate attachments

High
Pre-existing, amplifiedCost · PrivacyEffort MFix risk MediumConfidence High

Every ordinary message captures and sends the current PDF page, even for text-only questions. Explicitly attaching that same page adds another image. OpenAI/Gemini place all volatile text before the image, so changing conversation/request text prevents the image from being part of a reusable exact prefix across follow-ups. Tool-loop requests also resend the images.

AiPanel.swift:338–369 — captures a page image for every submission. AiStore.swift:388–392 — appends automatic and reference images without deduplication. OpenAIClient.swift:21–30 and GeminiClient.swift:30–35 — volatile joined text precedes images. OpenAI prompt caching guide — cache hits require exact prefixes, including image order/content.
Fix direction
Default to text-only. Send vision when the user attaches an image, the page has no text layer, or a lightweight intent rule identifies a visual question. Deduplicate images by digest. For implicit caching providers, order content as stable text → stable image → volatile request.

9. Send explicit snapshot references on OpenCode

High
IntroducedProvider parityEffort MFix risk LowConfidence High

The composer supports multiple region/page snapshots and the prompt says each is attached, but both OpenCode paths receive only context.currentPageImage. The actual reference images are silently omitted, so the model may analyze the wrong image while telling the user it saw the attachment.

AiPrompts.swift:118–130 — describes each reference image as attached. AiStore.swift:498–530 — OpenCode/Go pass only the automatic current-page image. OpenCodeZenClient.swift:49–58 — client accepts a single optional image.
Fix direction
Change OpenCode to accept a bounded image array, apply the same vision-capability gate, and keep prompt claims in sync with the images actually transmitted. Add parity tests for zero, one, and multiple attachments.

10. Avoid synchronous whole-document indexing on the main actor

High
IntroducedPerformanceEffort LFix risk MediumConfidence High

searchDocument waits for every missing page to be extracted before searching. PdfViewerController is main-actor isolated and calls PDFPage.string sequentially, yielding only every eight pages. Meanwhile a separate background-paced walk already indexes all pages, so the feature is API-on-demand but still eagerly consumes local CPU and memory after open.

AiToolEngine.swift:189–205 — full extraction precedes every broad search. PdfSelectionBridge.swift:591–605 — automatically walks all pages after load. PdfSelectionBridge.swift:617–645 — unpaced on-demand extraction remains on the main actor.
Fix direction
Build a persistent per-document text index keyed by file identity and modification time. Search indexed pages immediately, extract in cancellable bounded batches with progress, and avoid retaining duplicate full-document representations. Profile a 500+ page PDF before choosing batch size.

11. Complete OpenRouter cache routing and respect breakpoint limits

High
IntroducedCost · ReliabilityEffort SFix risk LowConfidence High

OpenRouterClient.generate receives the session id but never sends OpenRouter’s session_id, so changing first-message content can route follow-ups away from the warm provider. It also creates cache breakpoints on system, context, and every image; automatic image plus three snapshot references exceeds the documented four explicit breakpoints.

OpenRouterClient.swift:21–61 — session id is unused; each image gets a breakpoint. OpenRouterClient.swift:66–94 — request body has no session_id. OpenRouter prompt caching guide — recommends session ids for agent workflows and limits explicit Anthropic breakpoints to four.
Fix direction
Send a privacy-safe per-tab session_id. Reserve breakpoints for the largest stable prefixes; do not mark every image. Log cached/read/write tokens in a structured debug metric so the team can prove net savings.

12. Stop tool-loop exhaustion with an honest result

Medium
IntroducedCost · UXEffort SFix risk LowConfidence High

All providers can make eight separately generated rounds. If the model keeps calling rejected or repetitive tools, the loop exhausts and returns a synthetic “Done.” when any action result exists. The user sees success after a potentially expensive incomplete run.

OpenAIClient.swift:35–36, 118 GeminiClient.swift:55–56, 117 OpenRouterClient.swift:66–67, 172 — same pattern across clients.
Fix direction
Detect repeated identical calls and consecutive budget rejections, terminate early, and return a typed limit/interruption result. Preserve useful partial text and tell the user what could not be completed.

13. Remove the duplicated latest user request

Medium
Pre-existingCost · Prompt qualityEffort SFix risk LowConfidence High

The just-created user message is included in “Recent Conversation” and then repeated verbatim under “Latest User Request.” Every provider pays for it twice on every turn; repetition can also overweight the newest instruction.

AiStore.swift:374–386, 421–425 — conversation includes the new user message and passes the same trimmed request separately. AiPrompts.swift:52–58 — emits both sections.
Fix direction
Build recent conversation from messages preceding the current user message, or remove the separate latest-request section. Lock the intended shape with a snapshot test.

14. Make Keychain persistence transactional

Medium
IntroducedSecurity · ReliabilityEffort MFix risk MediumConfidence High

ChatGPT sign-in marks the account signed in even if storing tokens fails. Legacy key migration also uses one shared didMigrate flag: a different successful/empty migration can trigger a rewrite after one key failed to move, while the in-memory failed key has already been replaced with an empty Keychain read.

ChatGPTAuth.swift:207–224 — ignores the Keychain write result before setting signed-in state. AiPersistence.swift:46–83 — shared migration flag and immediate Keychain reload. AiPersistence.swift:85–103 — rewrites the blob based on later write results.
Fix direction
Return typed Keychain errors, verify the stored value before changing observable state, and migrate each legacy secret independently. Never remove a plaintext fallback until that exact secret has been read back successfully from Keychain.

15. Do not call “input-cheapest” the cheapest model

Medium
IntroducedCost UXEffort SFix risk LowConfidence High

The OpenRouter selector displays both input and output prices but its “Price” sort uses only prompt price. A model with cheap input and very expensive output can rank above a genuinely cheaper conversational model, contrary to the product goal.

ModelSelector.swift:495–527 — price comparator uses promptPrice only. ModelSelector.swift:584–604 — UI already has both prices.
Fix direction
Rename the sort to “Input price,” offer “Output price,” or use an explicitly documented blended estimate based on observed Vellum input/output ratios. Keep unknown/variable prices at the bottom.

Grounded direction

What to build next

1. A budget-aware retrieval layer

Separate retrieval state from chat history, return bounded chunks with coverage metadata, and persist a text index by document identity. This fixes both token economics and long-document trust.

2. A per-request cost envelope

Let Auto choose vision, reasoning, read count, and output budget from the request. Expose a small “Economy / Balanced / Deep” policy rather than independent knobs that can create invalid combinations.

3. Usage feedback users can act on

Capture provider usage fields—input, cached input, cache writes, output, and reasoning—then show a compact post-response estimate. The branch already requests OpenRouter usage but only logs it in debug.

4. Provider conformance fixtures

Define one internal turn model and fixture-test serialization for every provider. Provider-specific caches, signatures, tool calls, images, and truncation should live in adapters rather than six drifting loops.

What is working

Good decisions to preserve

  • The default prompt no longer dumps up to 120,000 characters of whole-document text every turn.
  • Stable-first prompt ordering and a privacy-safe per-tab OpenAI cache key are the right shape.
  • API keys move from UserDefaults toward Keychain, and OpenAI requests set store:false.
  • Streaming, activity states, model capability warnings, and clear-conversation cancellation improve user experience.
  • OpenRouter exposes both input and output pricing, free-only filtering, and capability-aware image/tool gating.
  • Search snippets are capped and regex evaluation is moved off the main actor with a deadline.
Cost addendum

Conversation and prompt caching

Removing raw retrieval output reduces billed input without sacrificing useful cache hits. Full page text remains in each provider’s in-memory tool loop for the request that needs it. The defect is copying that text into the persisted assistant message, where it becomes a large volatile tail on every later request. Removing it does not by itself raise the cache-hit percentage; the separate rolling-prefix changes below improve reuse. The best architecture is a stable prefix, a compact append-only conversation, and transient retrieval payloads.

What happens after the persistence fix

Still cached

  • System instructions and tool definitions.
  • Stable document/current-page context while it remains byte-identical.
  • Earlier items within a single multi-step tool request.
  • Compact user and assistant conversation turns.

No longer resent

  • Full text returned by getPageText.
  • Search snippets used only to produce an earlier answer.
  • Raw mutation/tool diagnostics embedded under “Actions”.
  • Irrelevant page text that would otherwise remain for ten messages.

High-leverage cache changes

C1. Preserve a genuinely append-only conversation prefix

High
IntroducedCost · Prompt shapeEffort MConfidence High

The branch serializes the current user message under “Recent Conversation” and then repeats it under “Latest User Request”. On the following turn, the earlier prompt diverges at that duplicate marker instead of remaining a clean prefix of the longer transcript. Removing the duplicate section—or representing turns as native message items—lets automatic prefix caching advance through prior conversation turns rather than stopping at the document-context boundary. The ten-message suffix(10) rollover and changes to current-page context will still reset longer prefixes; those are separate, expected boundaries that should be measured.

AiStore.swift:374–425 — the new user message enters messagesWithUser and is also passed as latestUserRequest. AiPrompts.swift:52–64 — both copies are serialized into the same volatile text block.
Fix direction
Prefer native provider message arrays: stable system/context first, then one item per user/assistant turn, with the newest user request appearing exactly once. If a shared fused prompt must remain, remove the duplicated latest-request section and add byte-prefix fixture tests across turns 1, 2, 3, and the ten-message rollover.

C2. Send the existing tab id to OpenRouter as session_id

High
Introduced gapOpenRouterEffort SConfidence High

sessionIdAtStart reaches OpenRouterClient.generate but is never placed in the request body or x-session-id header. OpenRouter documents session_id as the explicit sticky-routing key for multi-turn agent sessions; without it, provider stickiness depends on derived message hashes and may begin only after cache use is detected.

OpenRouterClient.swift:23–31, 67–92 — receives the tab id but omits it from the request.
Fix direction
Add a privacy-safe value such as "vellum-\(sessionIdAtStart)" as the top-level session_id. Keep it stable for the tab, rotate it for a new document session, and assert it in request-body tests.

C3. Stop treating every image as a cache breakpoint

High
IntroducedImages · OpenRouterEffort MConfidence High

OpenRouter currently adds one explicit cache_control breakpoint to the system prompt, one to document context, and one to every image. On Anthropic-routed requests, three images already produce five explicit breakpoints, beyond Anthropic’s four-breakpoint limit. Other routed providers interpret or ignore these markers differently. The automatically captured page image also changes with page/render state, so it can create an expensive cache write with little chance of reuse.

OpenRouterClient.swift:37–54 — one breakpoint per image. AiStore.swift:388–392 — automatic page image plus all referenced images.
Fix direction
Make page images on-demand. Reserve explicit breakpoints for the system prompt and one substantial stable context block; do not place a breakpoint on each attachment. Keep all provider requests within the strictest supported breakpoint budget.

C4. Measure cache economics in every provider path

High
Introduced gapObservabilityEffort MConfidence High

Only OpenRouter logs a final usage object, and only in debug builds. OpenAI/ChatGPT ignore completed-response usage, while Gemini ignores usage metadata. Without cache-read, cache-write, total-input, and model-price measurements, the product cannot tell whether a breakpoint or retention choice actually saves money.

OpenRouterClient.swift:107–113 — debug-only raw usage log. OpenAIClient.swift:67–87 — no completed-response usage handling. GeminiClient.swift:68–86 — candidate parsing ignores usage metadata.
Fix direction
Normalize provider usage into a local AiUsage record: input, cached-read, cache-write, reasoning, and output tokens plus estimated cost. Show a lightweight per-response cost/cache indicator and aggregate cache-hit rate by provider/model without storing prompt contents.

Caching ideas not worth shipping broadly

  • Do not persist raw retrieval text just to make it cacheable. A discounted unnecessary token is still more expensive than a token that is not sent, and large volatile tails crowd out relevant context.
  • Do not enable OpenRouter whole-response caching for general agent requests. This is a prospective warning—the branch currently implements prompt caching, not response caching. Exact repeated requests are uncommon in a conversation, and replaying a cached tool call could repeat a navigation or mutation. Restrict any future response cache to explicitly read-only, tool-disabled operations.
  • Do not cache the full PDF by default. The branch’s pull model avoids paying to send documents users never ask about. Cache only large stable context that has demonstrated repeated reuse.

Recommended execution order

  1. Stop persisting raw tool output and add a “retrieval marker never reaches the next prompt” test.
  2. Remove the duplicated latest request and add exact-prefix fixtures across multiple turns.
  3. Add OpenRouter session_id and enforce a maximum breakpoint count.
  4. Make images on-demand and keep volatile attachments after stable cache boundaries.
  5. Normalize usage telemetry, then decide retention and provider-specific caching from measured savings.
