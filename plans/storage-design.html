Vellum Storage Architecture

Design for Issue #29 · accounts for scratchpad (PR #44), ai-experience (PR #41), on-demand retrieval (PR #46 / Issue #37) · July 11, 2026 · revised July 14 after #41/#44/#45/#46/#48 merged to main
Revision status — July 14, 2026
All feature branches this plan accounted for have merged to main, and PR #48 shipped two pieces of this plan early, in slightly different form than designed:
  • Shipped: the persistent page-text cache (PageTextCache.swift, at App Support/text-cache/) — including this plan's key insight, hash-as-validator-not-key, refreshed after Vellum's own rewrites so annotating doesn't nuke the cache. Deviation: it keys by sha256(path) rather than a stable DocumentID (§3/§5 below, updated).
  • Shipped: Storage settings tab v1 — cache-only: total size, per-document rows with size/recency/completeness, per-row delete + erase-all, the exact "notes and conversations are NOT affected" copy, a never-auto-delete "original file not found" hint, and launch-time 6-month TTL eviction.
  • Shipped: conversations got a write-behind cache (in-memory entries, coalesced 200 ms background flush, awaited on quit) fixing the worst of the blob-rewrite cost — though still in UserDefaults, still path-keyed. Messages now also persist per-response token/cost usage.
  • Unchanged: scratchpad storage merged exactly as analyzed (UserDefaults blob, path keys, flat attachment pool, non-atomic writes); auto-archive still silently saves every opened page; no DocumentID anywhere.
§7's build plan is re-sequenced accordingly; the biggest remaining storage-growth hole is now the web archives (no size UI, no TTL, opening = permanent saving).
TL;DR. Vellum's best storage idea is already shipped: annotations live inside the document, so sharing a PDF shares the work. The problem is that everything the three in-flight branches add — scratchpad markdown, attachment images, AI conversations, extracted page text — is keyed by a raw file path and stored in UserDefaults or a flat app directory, so none of it survives a rename, move, or share. The fix is three moves, no backend, no accounts:
  1. One document identity — a /VellumDocId UUID stamped into the PDF's Info dictionary (the app already writes custom Info keys), so identity travels with the file. Web docs keep their existing URL hash.
  2. A per-document folder under a relocatable library root — documents/<docId>/ holding scratchpad.md, attachments/, conversations.json, meta.json — replacing the UserDefaults blobs. Derived data (page-text cache, auto-archives) moves to ~/Library/Caches where eviction is guilt-free.
  3. A .vellum export bundle — document + its folder zipped with the existing MiniZip/manifest machinery — for "share with notes." A bare PDF share still carries annotations for free.
  4. A Settings → Storage pane governed by one rule — nothing invisible: every persisted byte is either visible in its document's context or listed here with a delete button. Per-document size rows (notes / chat / cache / archive), safe "clear all caches," TTL eviction, and an orphaned-data section. Needed because auto-archiving currently grows storage ~2× per page ever opened.
This is the issue thread's Option C (local-first, user-relocatable, iCloud-ready later), extended to cover the new features and made rename/move/share-proof.

The identity problem underneath everything

The app uses three inconsistent identity schemes: PDFs are their canonicalized filesystem path (realpath), web docs are sha256(normalized URL), and every sidecar store (conversations, scratchpad, recents) keys on the raw DocumentInfo.pdfPath string. Nothing is content-addressed. Consequences:

  • Rename or move a PDF → its scratchpad, conversation history, and recents entry are silently orphaned. The data still exists, unreachable, counting against the 200/25-document eviction caps.
  • Share a PDF with someone (or with yourself on another Mac) → annotations arrive (embedded), but the scratchpad and chat stay behind — and could never re-attach anyway, because the recipient's path differs.
  • PR B's planned page-text cache wants content-hash keying — a fourth scheme — and a naive full-file hash breaks under Vellum's own design: every highlight atomically rewrites the whole PDF, so the hash churns on every annotation.

The efficiency problem

  • PR #44's scratchpad debounce re-encodes every document's note into one blob and rewrites it through CFPreferences on each 400 ms save. At the design caps that's a ~40 MB value serialized per keystroke-pause. Same pattern as conversations (which additionally hand-roll JSON key ordering to fake LRU semantics — ~30 lines of byte-scanning workaround that exists only because UserDefaults was the wrong container).
  • Attachment resolution is a directory scan per image load; GC is a global scan of all notes on every tab switch.
  • Fixed in #48: a 900-page PDF used to re-extract its text every open; PageTextCache now restores it from disk. Remaining alignment items: it keys by path (rename = orphaned cache entry until TTL sweeps it) and lives in App Support rather than Caches — both cheap to fix once DocumentID exists, since all keying funnels through one pathKey() function.
  • Derived, re-creatable data (auto-archived .vellumweb copies, future text cache) sits in Application Support next to irreplaceable user data, so nothing can ever be safely evicted and backups/sync carry dead weight.

2 · Data taxonomy — four classes, four homes

Every current and future piece of state falls into one of four classes. The rule: class determines home; no exceptions.

ClassExamplesHomeProperties
A. Document-embeddedPDF annotations, bookmarks, last page, /VellumDocId; .vellumweb contentsThe document file itselfTravels with the file, zero effort. Keep exactly as-is.
B. Document-adjacent user dataScratchpad text + attachments, AI conversations, web sidecar records, per-doc metadata<libraryRoot>/documents/<docId>/Irreplaceable. Syncable. Exportable in a bundle. One folder per document.
C. Derived / re-creatablePage-text cache (PR B), managed auto-archives, OpenRouter catalog, future OCR/RAG indexes~/Library/Caches/com.vellum.app/Evictable without loss. Never synced, never exported, TTL cleanup guilt-free.
D. Device-scopedAPI keys, OAuth tokens → Keychain; theme, font size, provider/model prefs, recents → UserDefaultsKeychain / UserDefaultsSmall scalars and secrets only. Never document data, never blobs.

PR #41 already put class D right (Keychain migration — keep it verbatim). PR #44 put class B data in class D's home; PR B would put class C data in class B's home unless redirected. This taxonomy is the one-line review criterion for every future feature: "which class is this, and is it in that class's home?"

3 · Document identity: DocumentID

Decision
A single DocumentID string used by every class-B and class-C store, resolved per document kind:
  • Web: the existing sha256(normalized URL) — unchanged, byte-compatible with the Tauri-era library, nothing re-keys.
  • PDF: a UUID stamped once into the Info dictionary as /VellumDocId — the same mechanism as the existing /VellumLastPage. The ID lives in the file, so move/rename/copy/AirDrop all preserve it, and a recipient's Vellum recognizes the document.
  • Read-only PDF fallback (no write permission / locked): sha256 of file bytes. Safe precisely because Vellum can't rewrite a file it can't write to, so the hash is stable. Final fallback: canonical path (today's behavior).

Stamp lazily — never touch a file the user hasn't invested in

Do not stamp on open. Stamp on the first write Vellum was going to make anyway — first annotation, first metadata save — or on the first creation of class-B data (first scratchpad keystroke, first AI message). Until then, nothing needs a durable identity. This avoids "I opened a PDF and the app modified it," avoids an extra multi-MB atomic rewrite on every open, and piggybacks the stamp on an existing rewrite so it costs nothing.

Copies share an ID — that's a feature

Duplicating a PDF duplicates its ID, so both copies resolve to the same documents/<docId>/ folder: your notes follow the document, not the path. This is the behavior a user who reorganizes folders actually wants. (If true fork-the-notes semantics are ever needed, "Duplicate as new document" can re-stamp — don't build it until asked for.)

Cache keying: half of this shipped in #48

The design called for the text cache to be keyed by DocumentID (stable under Vellum's own annotation rewrites) with a content hash used only as a validation stamp, refreshed when Vellum itself rewrites the PDF and invalidating only on external edits. The shipped PageTextCache implements the stamp mechanism exactly (partial hash over size + first/last 4 MB; refreshHash called by PdfDocumentIO after in-app writes) — independently arriving at the same reasoning, documented in its header comment. What it lacks is the stable key: it keys by sha256(path), so a renamed PDF orphans its cache entry (shown as "original file not found" in Settings, swept by TTL, rebuilt from scratch at the new path). When DocumentID lands, re-keying is mechanical: all keying funnels through PageTextCache.pathKey(), and the validator logic transfers unchanged.

Why files, per document, instead of one database or one blob:

  • Write cost is proportional to the edit. A scratchpad keystroke rewrites one small scratchpad.md, not an all-documents blob through CFPreferences. Use the existing tmp-file + rename(2) pattern (WebLibrary.saveRecord already does this) — PR #44's plain Data.write attachment saves should adopt it too.
  • Sync conflict granularity = one document. Two Macs editing different documents never conflict; same document → last-writer-wins on a per-file basis, which is the acceptable worst case the issue thread already accepted for annotations.
  • Human-legible and tool-friendly. scratchpad.md with relative image paths opens in Obsidian/any editor; the folder is greppable and Spotlight-indexable. (The editor's vellum-scratchpad:// scheme handler becomes a load/save-time rewrite to relative paths, or resolves relative paths directly — implementation detail, but persist the portable form.)
  • GC becomes trivial and correct. Attachments are scoped per document: prune = scan one note, not all 200; delete a document's data = delete one folder. The O(n) global directory scan and cross-document GC in PR #44 disappear.
  • Eviction caps become unnecessary. The 25-conversation / 200-note LRU caps exist because a single plist value can't grow unbounded. Per-document files can just… exist. Settings can show per-document sizes (PR B already planned this UI for the cache — same list, one more column).

The auto-archive catch: today, opening = saving

The layout above assumes a distinction between pages the user chose to keep (web/) and archives created as a side effect (Caches). That distinction does not exist today: 1.5 s after any webpage loads, archiveDefault (WebSessionBackend.swift:225, triggered from WebViewerView.swift:896) re-fetches the page, writes the .vellumweb and the unpacked archives/<key>/ mirror (~2× the bytes), and calls markSavedIfAbsent — so every page ever opened is silently promoted to "saved" and its archives are permanent. A user who reads widely accumulates storage they never asked for and can only shed one page at a time.

Recommendation
Make saving explicit (a save button/toggle, as the sidecar's saved flag was clearly designed for) and demote on-open archives to class C: they live in Caches, still give instant offline reload and AI text, but are TTL-evictable and deletable in bulk from the Storage pane. Explicitly saved pages get the durable web/ treatment. Annotating a page should auto-promote it to saved — annotations are user investment. If instead "everything I open is my library" is the intended UX, the Storage pane below still keeps it manageable — but the default should not be permanent either way.

TierWhat the user doesWhat travelsCost to build

2. .vellum bundle — "Export with notes…"One zip: the document + manifest.json + scratchpad.md + attachments/ + (opt-in checkbox) conversations.jsonEverythingSmall — reuse MiniZip, the manifest/sha256-verification pattern, and zip-slip guards from WebArchive.swift wholesale
3. Synced libraryPoint libraryRoot at iCloud Drive / Dropbox in SettingsAll class-B data across the user's Macs, continuouslySettings picker + one-time copy migration (the issue thread's Option C, ~150 lines)

On import of a .vellum bundle: verify hashes, write the document where the user chooses, unpack the sidecar into documents/<docId>/ (merge-prompt if the ID already exists locally). Because identity is embedded in the PDF, the recipient's app links everything with zero configuration. Conversations default to excluded from export — they're chat logs, often semi-private, and the checkbox makes sharing intent explicit.

Considered and rejected — embedding the scratchpad inside the PDF itself (PDF /EmbeddedFiles attachments): maximal portability, but every debounced scratchpad save would atomically rewrite the entire (possibly 500 MB) PDF, images bloat the document permanently, other viewers surface mystery attachments, and web documents have no equivalent. Wrong write-amplification trade for live data; tier 2 gets the same result at export time only.

6 · Sync & backend posture

Local-first, no accounts, no server — confirming the issue thread's recommendation, now with more evidence:

  • Supabase/backend: rejected. The document model is file-embedded by design; a central annotation database fights the architecture's best property. Accounts are friction for a reader app; you'd be hosting user data for no feature gain. Revisit only if collaboration/multi-user ever becomes a goal.
  • SQLite/Core Data: rejected for class B. A single DB file is the worst possible sync-conflict unit, is opaque to users, and buys nothing at this scale (a power user has hundreds of documents, not millions of rows). Class C caches may use whatever format is convenient — they're disposable.
  • iCloud: deferred, path preserved. Tier 3 gives motivated users sync now with any provider. When native sync is worth the entitlement + testing overhead, swap libraryRoot for the ubiquity container and add NSFilePresenter observation — the per-document-folder layout is exactly what iCloud Drive syncs well. Nothing in this design has to be redone.

7 · Migration & sequencing (re-sequenced July 14 — all feature PRs merged; PR B + Storage tab v1 shipped in #48)

Storage PR 0 — web-archive management + explicit save new · next up

With the text cache shipped and UI-managed, the web archives are now the only storage surface that grows unbounded with zero visibility: every opened page permanently stores a .vellumweb + an unpacked mirror (~2×) and is silently marked saved. Independent of DocumentID, so it can ship first:

  • Decide the §4 question (recommendation: explicit save; annotating auto-promotes; on-open archives become evictable, TTL'd like the text cache).
  • Extend the shipped Storage tab with a Web pages section: per-page rows (title, size of record + archives, last opened, saved vs auto), row delete, bulk "remove archives older than…". Reuses the exact row/dialog/TTL patterns StorageSettingsTab and PageTextCache.evictStale just established.

Storage PR 1 — identity + document store (the core)

  • DocumentID resolver (web hash / /VellumDocId read + lazy stamp / byte-hash fallback / path fallback).
  • DocumentDataStore: the documents/<docId>/ layout, atomic writes, one facade API that ScratchpadStore/ScratchpadPersistence and AiPersistence retarget to. Both already isolate persistence behind small enums, so this is a backend swap, not a refactor. Preserve #48's semantics in the swap: the in-memory-cache + coalesced-flush + await-on-quit contract for conversations (simpler now — flushing one document writes one small file, not the cross-document blob), and round-trip the new per-message usage field.
  • Re-key PageTextCache from sha256(path) to DocumentID (single pathKey() chokepoint; validator untouched) and optionally relocate text-cache/ to ~/Library/Caches per the §2 taxonomy — a one-time move, since a cold cache just rebuilds.
  • Cleanups inherited from the merged scratchpad PR: tmp+rename for attachment writes; persist markdown image refs as relative attachments/… paths (so a document's folder is portable standard Markdown).
  • Lazy migration, not big-bang: legacy UserDefaults blobs stay as a read-fallback. When a document is opened and acquires its ID, its scratchpad + conversation entries migrate into the folder and are removed from the blobs. No mass PDF-stamping pass, no touching files at migration time. After ~a release cycle, an idle-time sweep migrates leftovers whose paths still resolve; truly orphaned entries surface in Settings for export-or-delete.
  • Attachment ownership for migration: the markdown refs in each note identify which document owns which attachment — move files into the owning folder during that document's lazy migration.
  • Recents gain a docId field beside the path: when the path 404s, re-resolve by ID before giving up (fixes the silent dead-recent today).

The page-text cache, TTL eviction, and the cache section of the Storage pane all shipped in PR #48 (see revision status at top). The two deviations from this plan — path keying and the App Support location — are absorbed into Storage PR 1 above as small alignment items. The archive sections originally scoped here moved up to Storage PR 0.

  • .vellum bundle export/import (tier 2).

8 · Storage management UI — the "nothing invisible" rule

Guiding principle (owner decision, July 11 2026): the app must not persist anything the user cannot see and delete. Storage that only ever grows, in places only Finder archaeology can find, is a bug even when each individual write is justified. Two consequences:

  • Delete means delete. The chat panel's clear button already hard-deletes the persisted conversation — clearConversation() (AiStore.swift:135) saves an empty list and AiPersistence removes the entry outright. The new DocumentDataStore must preserve this as a contract: clearing chat deletes conversations.json; clearing the scratchpad deletes scratchpad.md and its now-unreferenced attachments; a folder with no remaining files is removed. Same for the Storage pane's row actions.
  • No invisible hoards. Today's design violates the rule twice: conversations are retained for up to 25 documents but no UI lists them — only the active document's history is ever visible — and path-keyed entries orphaned by a rename can never be seen again by anyone. The DocumentID layer fixes the orphaning; the Storage pane fixes the visibility.

Settings → Storage pane

One pane covering all four data classes. v1 shipped in #48 (StorageSettingsTab): the extracted-text-cache sections — total size, per-document rows with size/recency/completeness, per-row delete and erase-all with exactly the reassurance copy specified below, a "original file not found" hint that never auto-deletes, and a fixed 6-month launch-time TTL. 9 · Open questions

  • Explicit save vs. everything-saved for webpages — §4's recommendation (explicit save; on-open archives become evictable cache; annotating auto-promotes). Owner call on the UX; the Storage pane works under either.
  • Conversations in bundles — default-off checkbox proposed above; confirm.
  • Sandboxing — the app is currently unsandboxed (no entitlements file), which is why bare paths work. If Mac App Store distribution ever matters, recents/meta.json paths need security-scoped bookmarks; the DocumentID layer makes that an additive change, but decide before building tier-3 sync UI.
  • .vellumweb v2 — fold scratchpad.md + attachments into the archive itself (bump manifest version) vs. always using the .vellum wrapper for web docs too. Wrapper-always is simpler; one format fewer to version.
  • USER.md / SOUL.md caps — the Hermes-style bounded-write policy from issue #37 (per-write character budget, app-level file) — spec when that feature is picked up; the layout already reserves ai/.
