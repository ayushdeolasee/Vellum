### Plan 004 — web bridge isolation (reviewed)

Diff is +135 / −12 across 5 files. Verified independently: 38/38 tests,
`BUILD SUCCEEDED`. The isolation probe genuinely passed against a live app —
`window.webkit.messageHandlers.vellum` threw in the page world, the probe page's
title read `BRIDGE-ISOLATED`, and the forged `navigate` message was ignored. The
offline/snapshot path also re-anchored a bookmark, proving the `vellum-offline`
meta tag is read from inside the private world.

The executor caught something the plan didn't specify: `removeScriptMessageHandler`
must **also** name the content world, or teardown silently fails to remove the
handler. Good catch.

**Accepted regression → plan 008.** The content script overrides `window.open`
(`WebContentScript.swift:1477`) to route JS-initiated navigation back into the
reader. That override now patches only the *bridge world's* `window`, so page-JS
`window.open(...)` calls no longer reach the app. Link **clicks** are unaffected
(DOM events cross worlds fine) — this only affects JS-initiated opens. Restoring it
needs a `WKUIDelegate` (`createWebViewWith:` → `post("navigate")`), which is new
behavior and out of 004's scope. Tracked as plan 008; the security win lands now,
parity follows.

**A red herring worth recording**, because it nearly cost us plan 005: the executor
observed that a JS-written `document.title` did not reappear on offline reload and
inferred that page JS "already doesn't execute" in snapshots. It does. The reason
the title didn't reappear is that `sanitizeSnapshotHtml` strips `<script>` **tags**
at capture — which says nothing about inline `onerror`/`onload` handlers,
`javascript:` URIs, or SVG scripting, none of which are script tags and none of
which are stripped today. Plan 005's dispatch carries an explicit warning not to
skip either layer on the basis of that observation.

Minor: `WebHtml.jsonString` is now dead (its only caller was the deleted injection).
Left in place to keep the diff scoped; trivial to remove later.

### Plan 002 — annotation save errors (reviewed)

Diff is +191 / −0 across 4 files: purely additive, as intended. `saveError` /
`reportSaveError` added to `AnnotationStore`; every existing `NSLog` and every
rollback branch left byte-identical (verified in the diff). Banner lives in
`PaneView` as a top-aligned overlay with `.allowsHitTesting(false)`, so it cannot
swallow clicks meant for the selection popover or sticky-note overlay underneath.

Test audit (the thing executors most often fake — these are real): a
`FakeSessionService` conforming to the existing `SessionService` **protocol** is
injected through `AppStore`'s existing `init(sessions:)`, so no production
initializer changed. Each test injects a genuine write failure and asserts **both**
the user-visible `saveError` **and** the correct rollback — failed create rolls the
optimistic row back, failed update reloads from the backend, failed delete restores
the previous list. The create test synchronizes with the detached persist `Task` by
going through the store's own `pendingCreates`/`awaitPendingCreate` mechanism rather
than sleeping — no flake.

One done-criterion was written wrong by me, not missed by the executor: I asked for
exactly 1 `grep` match on `annotations.saveError` in `PaneView.swift`; rendering it
conditionally *and* tagging it with the accessibility identifier necessarily yields
3. Accepted.

GUI spot-check declined, correctly: another Vellum instance (pid 86653) was running
outside the worktree, and driving a canvas drag would have risked targeting the wrong
instance. The three failure paths are covered more precisely by the unit tests anyway.

  trunk; see the status table.
