PR B — Persistent Page-Text Cache + Cache Management

Vellum · issue #37 · branch ai-ondemand-retrieval · plan synthesized 2026-07-10 from Claude Fable (lead), two Opus 4.8 design agents, and GPT-5.6 Sol (codex) — all grounded in the live tree at commit 33df379.

Goal. PR A made the AI agent read documents through an in-memory page-text store (AiStore.pageTexts) that is rebuilt from scratch on every open. PR B persists that extracted text per PDF so reopening is instant (the .indexing pill should effectively never appear for a previously-opened document), adds a Storage settings tab (sizes, per-document delete, erase-all), and evicts caches not opened in ~6 months at launch. All commits land on this same branch/PR.

2 · The contested decision: cache identity

The spec says "key by content hash so external edits invalidate." The complication: PdfDocumentIO rewrites the PDF on every annotation create/update/delete and on setMetadata("last_page"), which fires for every tab at quit (VellumApp.swift:17). Three options were on the table:

OptionProposed byVerdict
Hash once at open; accept a miss after any mutating sessionOpus #1Rejected — quit itself mutates, so nearly every reopen would miss; the cache would be near-useless.
Hash-keyed files + alias table re-registered around each rewriteSolCorrect but heavy — alias indirection, orphaned files, rename races.
Path-keyed files, hash stored inside the index entry for validation, refreshed after each in-app writeFableChosen
Chosen scheme. Cache file = text-cache/<sha256(canonicalPath)>.json (same keying style as WebLibrary.pageKey; conversations are already keyed by pdfPath, so identity-by-path is idiomatic here). The index entry stores a contentHash. On open, the hash of the just-read Data is compared against the entry: match → instant restore; mismatch → external edit, discard and re-extract. After every successful in-app write, PdfDocumentIO hashes the just-written Data (already in memory, off-main on that actor) and refreshes the stored hash inline before returning — the IO actor serializes writes per document, so there is no visibility race, and the quit path (applicationShouldTerminate awaits setDocumentMetadata) completes the refresh before termination. External edits still invalidate; in-app writes no longer do. No renames, no orphans, no alias table.

Hash refresh before first flush: the cache actor keeps an in-memory latestHash[path] updated by refreshHash even when no index entry exists yet; a later flush stamps the entry with that value, so "annotate before the first 50-page flush" cannot poison the entry with a stale hash.

Hash input: SHA256 over (UInt64 LE file size + first 4 MB + last 4 MB) — constant ~8 MB cost regardless of file size, effectively a full hash for files < 8 MB, spec-sanctioned; collisions merely show stale cache text and self-heal on the next write. File format: one JSON per document {version, pageCount, complete, pages:{"1": "…", "2": ""}} — empty string is meaningful (scanned page, kept for the future OCR pass) — rewritten fully per debounced flush via tmp+rename(2) (the existing WebLibrary/PdfAtomicWriter pattern; ~1-2 MB per rewrite, so JSONL append complexity isn't warranted).

4 · Storage settings tab (Opus #2's design, adopted)

  • Fifth tab, Label("Storage", systemImage: "internaldrive"). Grouped Form, scrolling enabled (unlike siblings) with a bounded .frame(height: 460); rows are a plain ForEach in a Section — never a List nested in a Form.
  • Summary section: total size (.byteCount(style: .file), Finder-style) + destructive "Erase all…". Rows sorted by size desc: middle-truncated title, relative last-opened (.relative(presentation: .named)), status — Complete / Partial / gold "Original file not found" hint (display-only; never auto-deleted).
  • Both deletes confirm via confirmationDialog. Copy (verbatim): "This removes only the extracted-text cache Vellum uses to speed up AI and search. Your notes, highlights, and AI conversations are not affected. The text is rebuilt automatically the next time you open this document." (erase-all variant covers every document).
  • Sizes and sourceExists resolved inside the actor at listEntries() time — no FileManager on the main thread. Optimistic local row removal after delete; full reload only on tab appear.
  • Deleting the open document's cache is allowed and benign (in-memory text persists and re-writes on next flush) — no special-casing; the Settings scene doesn't even have AiStore injected.
  • Accessibility identifiers, dot-namespaced per house convention: storageRow.<key>, storageRow.delete.<key>, storage.eraseAll, storage.confirmDelete, storage.confirmEraseAll, storage.empty — these drive the macos-app-driver verification.

5 · Edge cases & risks (union of all three reviews)

RiskMitigation
Hash churn from in-app rewrites (Sol's #1, Fable's finding)Path-keyed entries + inline hash refresh in PdfDocumentIO after every save (§2).
Main-thread stalls (previously shipped bug class)Hashing, JSON codec, sizing, deletes, eviction all inside PageTextCache; only page.string and dict mutation stay on main. Bulk restore is one dict merge.
Empty text treated as uncached → scanned pages re-extracted forever, complete never truePresence-of-key semantics everywhere; "" round-trips; completion = key coverage of 1…N.
Stale writes crossing a tab switch; clearDocumentContext called from two lifecycle sitesactiveTabId == tabId guard after every await in load() (existing pattern); persister owns its data and is flushed before resets.
Flush racing delete/eviction; truncated filesAll disk mutations serialize through the one actor; tmp+rename writes; corrupt/version-mismatched JSON decodes as a miss and regenerates (matches AiPersistence's silent-recovery style).
0-page docs / two tabs same pathNo persister below 1 page; AppStore already prevents duplicate same-path tabs, and same-path writes serialize through the shared actor regardless.
