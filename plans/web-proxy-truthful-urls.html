Truthful proxy URLs for Vellum’s web viewer

Plan v2 — 2026-07-12 · branch ai-ondemand-retrieval (fix is branch-independent) · reviewed adversarially by three independent Fable agents (codebase-correctness, WebKit-empirical, regression/migration); all findings incorporated. Status: implemented & QA-verified 2026-07-12 (headline bug fixed; user opted to drop the legacy ?url= route entirely — pre-ship, no compat needed. Offline snapshot-only render still unverified live: fresh QA launches were blocked by a keychain prompt from a synchronous main-thread SecItemCopyMatching in app init — separate bug).

1 · The bug

Opening https://www.anthropic.com/research/global-workspace in Vellum shows WebKit’s native “This page couldn’t load” screen (or a blank page). Confirmed by live QA reproduction: Vellum’s fetch of the page succeeds (the on-disk snapshot was rewritten with the exact 216 KB live body during the failure); the page destroys itself after rendering.

Vellum serves every live page from vellum-web://localhost/?url=<encoded> (VellumWebSchemeHandler, WebPageExtractor.swift:686), so window.location.pathname is always /. Client-side-routed sites (anthropic.com is Next.js App Router — __next_f flight data in the HTML) hydrate by comparing their server-rendered route to window.location, see a mismatch, and tear down: blank page, History-API URL rewrites that drop ?url= (reload → the handler’s “Missing url parameter” 404), failed navigations → WebKit’s error page. The homepage works only because its real path is /; example.com works because it has no client router. Inherited 1:1 from the Tauri port — not a regression of this branch.

2 · The fix in one line

Serve pages under a URL whose host, path, and query are real:

https://www.anthropic.com/research/global-workspace
        ↓
vellum-web://www.anthropic.com/research/global-workspace

http:// targets use a second registered scheme, vellum-webi:// (“i” = insecure), since the authority has no room for the real scheme. Ports are preserved (vellum-web://localhost:3000/dev-page).

3 · Empirical validation (WebKit probe, this machine, macOS 27)

A reviewer agent built a live WKWebView harness (/tmp/vellum-scheme-probe/) rather than trusting the URL spec. Results:

  • PROVEN Custom-scheme URLs parse hierarchically: location.pathname/search/host/port all report real values; origin is a per-host tuple origin (not "null"); percent-encoding reaches the scheme handler verbatim. The core mechanism works.
  • BLOCKER FOUND With the injected <base href="https://…"> present, a relative history.pushState/replaceState resolves against the base → https URL → cross-origin SecurityError. Next.js hydration calls exactly this. Resolved by work item 4 (History API shim) — verified end-to-end in the same probe.
  • PROVEN <base> routes relative and scheme-relative (//cdn…) subresources to https — no regression.
  • FOUND WebKit fires Link: rel=preload headers from cross-origin subresource responses as phantom requests at the scheme handler (anthropic’s 404 page carries 18 font preloads). Resolved by work item 3 (main-document guard), which also closes a request-amplification vector.
  • PROVEN Dual scheme registration fine; .invalid reserved host fine; localStorage now works per-host (today: one shared origin for all sites — the change fixes cross-site storage leakage); cookies remain a silent no-op (status quo); service-worker registration fails gracefully (status quo); fetch sends Origin: vellum-web://<host>, CORS behavior unchanged.

4 · Work items

All in Vellum/Services/Web/WebPageExtractor.swift, Vellum/Views/Web/WebViewerView.swift, Vellum/Views/Web/WebContentScript.swift, plus tests. Storage formats, keys (sha256(normalized URL)), archives, and the AI stack are untouched (verified: no on-disk artifact encodes the serving origin — no data migration).

  1. Proxy URL builder. proxyUrl(for:) (WebPageExtractor.swift:686): map the normalized target to vellum-web(i)://host[:port]path[?query] by string surgery on the already-WHATWG-serialized normalized URL (no decode/re-encode). Register vellum-webi alongside vellum-web in makeWebView (WebViewerView.swift:274).
  2. Scheme-handler routing (handleRequest, WebPageExtractor.swift:723):
    • Archive assets move to a reserved host: vellum-web://assets.vellum.invalid/<key>/<name>. The request-URL-derived assetBase computation (:753–760) is deleted and replaced by this constant on all three serving paths (snapshot-only, offline fallback, live). Asset responses gain Access-Control-Allow-Origin: * (page and assets no longer share an origin).
    • Page requests: reconstruct https:// (or http:// for vellum-webi) + host[:port] + URLComponents.percentEncodedPath + percentEncodedQuery, then WebUrl.normalize → existing fetch/prepare/serve pipeline unchanged.
    • Legacy ?url= route: dead code — the webview URL is never persisted; restore always re-derives from the stored real URL (WorkspaceService.swift:12, AppStore.swift:311, WebViewerView.swift:324). Kept as a ~5-line belt-and-braces branch (host localhost, no port, path /, has url= pair), covered by unit test only.
  3. Main-document guard. Serve the page pipeline only when request.url == request.mainDocumentURL; answer any other real-host request (Link-preload phantoms, stray subresources) with 204 and no network fetch. Closes the amplification vector where any site’s response headers could make Vellum issue arbitrary fetches.
  4. History API shim in the content script (new code — there is no history interception today; all 1828 lines checked). Injected before site scripts run: wrap history.pushState/replaceState to (a) resolve relative URL arguments against location.href instead of the <base>, and (b) map absolute https://<this-host>/… arguments onto the vellum-web origin. Without this, Next.js hydration throws SecurityError (empirically proven) and the headline bug survives. The shim also updates PAGE_URL and posts a lightweight url-changed message so soft navigations no longer mis-attribute auto-archives and annotations to the mount URL.
  5. Redirect rebind must reload. Today a server redirect serves the destination’s HTML under the original request URL and the shell rebind (handleInit, WebViewerView.swift:890–904) never reloads the webview — under truthful URLs that recreates the exact path-mismatch bug for any 301 (trailing slash, locale, renamed article). When init reports an effective URL whose host/path differ from the serving URL, reload proxyUrl(effectiveUrl) with a one-shot guard against redirect loops.
  6. Navigation delegate (mandatory, not belt-and-braces). The webview currently has no WKNavigationDelegate; because <base> makes missed links resolve to https, a router’s location.assign today escapes the reader entirely into the raw site. Add:
    • decidePolicyFor: cancel main-frame-only (targetFrame?.isMainFrame == true) http(s) navigations and route them through the existing navigate/rebind flow. Subframes (article embeds, YouTube iframes) pass through untouched.
    • createWebViewWith (UI delegate): route window.open/target=_blank into the same flow instead of silently dropping them.
    • didFailProvisionalNavigation/didFail: if a snapshot exists for the current document, serve the offline snapshot (existing prepareHtml(…, offline: true) path via a reserved snapshot route); else Vellum’s own error page — never WebKit’s.
    • webViewWebContentProcessDidTerminate: one reload, then snapshot fallback.
  7. Tests (note: advisor-plans/001 says the test target needs fixing first — land that or run these as a temporary standalone target):
    • Round-trip: normalized URL → proxyUrl → handler reconstruction → normalize == identity, over ports, encoded segments, punycode hosts, userinfo, IPv6 literals, http vs https, root path, trailing slash, query with =/& values.
    • Legacy route resolves; real https://localhost/page?url=x is not misrouted into legacy handling.
    • Served snapshot HTML references only assets.vellum.invalid asset URLs (guards work item 2 against partial application).
    • Real-site /asset/… path is fetched as a page, not swallowed by the archive route.

6 · Non-goals & accepted risks

  • No move to direct https browsing + WKUserScript: the probe results remove the need — the proxy architecture works with truthful URLs.
  • location.protocol/origin still aren’t https: — frameworks that branch on protocol may still misbehave; only live QA settles residual cases. Cookie-gated hydration still won’t work (cookies are a no-op on custom schemes, unchanged).
  • Sites hard-requiring service workers degrade exactly as today.
  • YouTube embeds cannot play in the reader (discovered post-ship): YouTube’s embed player hard-requires an http(s) Referer, which WebKit never sends from a custom-scheme document — every variant (bare, autoplay, jsapi+https-origin) returns “Error 153: Video player configuration error”; rewriting the origin query param does not help (all verified empirically). Mitigation shipped: the content script replaces YouTube embed iframes with a thumbnail facade (i.ytimg.com poster + play badge) that opens the video in the system browser via a new open-external message.
  • Docs cleanup (specs/SPECS-web.md etc. mention vellum-web://localhost) is optional follow-up.
