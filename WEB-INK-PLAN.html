Apple Pencil ink on web pages

Draw with Apple Pencil directly on web pages in the iPad reader, with the same tool palette, feel, and persistence guarantees as the existing PDF ink feature. Polish bar is GoodNotes: PencilKit-native latency, pressure, tilt, crisp strokes at zoom, no lost ink ever.

1Why this is genuinely hard (read this first)

The PDF ink stack works because PDF pages are fixed geometry: a page is always 612×792 pt, PDFKit tells us exactly where each page sits on screen (PDFPageOverlayViewProvider), and a stroke stored in page coordinates is valid forever. None of that is true for web pages:

  1. No pages, one giant scrolling document. The web view is a single continuous DOM document that scrolls via WKWebView's native UIScrollView. Vellum's "pages" are virtual (~3600-char text chunks) — text constructs, not geometry.
  2. Layout is not stable. Pages reflow after load (hydration, lazy images, web fonts), on rotation/split-screen (viewport width change re-wraps every line), and on SPA re-renders. A stroke stored at y = 14,200px points at different content after any of these. The existing highlight system dodges this by storing no geometry at all — highlights are text-quote anchors (offset + prefix/suffix) fuzzily re-resolved by JS after every reflow (resolveHighlight, WebContentScript.swift:654). Ink is pure geometry with no text to re-anchor to. Bridging that gap is the core design problem.
  3. Two independent zoom systems. Toolbar zoom is webView.pageZoom (CSS zoom — changes layout size). Native pinch is scrollView.zoomScale / VisualViewport (visual magnification — layout unchanged). The overlay must track both (WebViewerView_iOS.swift:1219-1224 already composes app.zoom * visualScale for popover math).

2Approaches considered

A · JS canvas inside the DOM Rejected

Draw via Pointer Events into a <canvas> element. Perfect scroll/reflow anchoring potential, but loses everything that makes Pencil feel right — PencilKit's Metal-latched low latency, predicted touches, pressure/tilt ink models, the eraser modes, system undo gesture, and the entire existing tool palette. Fails the GoodNotes bar.

B · Freeze the page to PDF Rejected as primary

WKWebView.createPDF, then reuse the PDF ink stack wholesale. Maximal reuse and layout stability, but kills the live page (links, video, find; existing text-anchored highlights live in a different document identity). Could become a future "annotate as PDF" export mode.

C · Native PencilKit overlay above the WKWebView, scroll-synced Chosen

Full reuse of the proven ink feel + palette; the cost is that we own scroll/zoom/reflow synchronization — which is tractable, as the design below shows.

3Design decisions

  1. One viewport-sized PKCanvasView, not per-page/banded canvases. PKCanvasView is a UIScrollView: give it contentSize = the full web document size and sync its contentOffset with the web view's. PencilKit tiles its rendering internally (this is how Notes does unbounded canvases), so a 200,000 pt-tall drawing surface is fine — we never create a giant UIView. One canvas ⇒ one PKDrawing, one undoManager, no stroke-crosses-band problem. The canvas mounts as a sibling view above the WKWebView inside WebViewRepresentable_iOS's container (below the SwiftUI popover layer).
  2. Scroll sync is bidirectional, mode-dependent, via KVO.
    • Ink inactive: canvas isUserInteractionEnabled = false (all touches pass to the page, exactly like PDF's applyPolicy); KVO on webView.scrollView.contentOffset drives canvas.contentOffset in the same runloop turn — no visible lag.
    • Ink active: the canvas sits on top and becomes the scroller. With drawingPolicy = .pencilOnly, pencil draws and fingers pan the canvas's own scroll view; we mirror canvas.contentOffset back into webView.scrollView.contentOffset (which fires the page's JS scroll events naturally, so virtual-page tracking / reportScroll keep working untouched).
    • An isSyncing reentrancy flag prevents feedback loops. Nobody replaces the web view's scrollView.delegate (WebKit owns it) — KVO only.
  3. Stroke coordinate space: zoom-1 CSS-pixel document space. Everything persisted is normalized to pageZoom == 1. At runtime the canvas operates in "zoomed document space" (CSS px × pageZoom): when toolbar zoom changes we rescale the live PKDrawing by newZoom/oldZoom and re-apply the tool at the scaled width — the exact setSuperSample/scaleDrawing trick from InkOverlayProvider_iOS.swift:218-234, which also means strokes re-rasterize at the new size and stay crisp at zoom for free (toolbar zoom is discrete, so no mid-gesture rescale problem).
  4. Native pinch (zoomScale) is view-transform-only, never folded into stored coordinates. KVO on scrollView.zoomScale + contentOffset applies a matching CGAffineTransform to the canvas so ink tracks the pinch (briefly bilinear-scaled mid-gesture, same as PDF mid-pinch). Deep-zoom crispness via a super-sample bump after pinch settles is a Phase 5 enhancement, not MVP.
  5. Persistence: a separate <key>.ink.json sidecar — NOT a field inside WebPageRecord. Critical cross-device gotcha: the macOS app re-encodes <key>.json on every annotation edit with its own Codable model; Codable silently drops unknown fields on encode, so an ink field added only on iPad would be deleted by the Mac the next time it touches a synced record (iCloud storage mode). A separate file the Mac never opens is immune. 
    {
  "version": 1,
  "url": "<normalized url>",
  "updated_at": "<rfc3339>",
  "layout": { "content_width": 980, "doc_height": 42000 },
  "clusters": [
    {
      "id": "<uuid>",
      "drawing": "<base64 PKDrawing, zoom-1 CSS px, cluster-local origin>",
      "bounds": { "x": ..., "y": ..., "w": ..., "h": ... },
      "anchor": {            // Phase 3; null in MVP
        "start_offset": 18234,
        "prefix": "…", "suffix": "…",
        "rect": { "x": ..., "y": ..., "w": ..., "h": ... }
      }
    }
  ]
}
    Clusters are derived data: at save time, strokes are grouped by spatial proximity from the single live PKDrawing; at load time, cluster drawings are translated (per re-anchor) and merged back into one drawing. MVP writes one cluster with a null anchor; the format doesn't change when Phase 3 lands.
  6. Write path mirrors PDF ink's hard-won rules. 700 ms per-document debounce; atomic tmp→rename; all I/O on an actor off the main thread (WebInkIO, following WebDocumentIO's pattern in WebSessionBackend.swift:328); a flushPendingInkAndWait() called when ink mode turns off, on tab close, and from the same scene-background hook the PDF flush uses (last stroke can never be lost to suspension). Per-path NSLock via the WebLibrary.recordLock pattern so ink writes can't race the record writes.
  7. First stroke promotes the page to saved. Same auto-save-on-annotate behavior as WebDocumentIO.createAnnotation (WebSessionBackend.swift:347), so an inked page always has an offline snapshot — the layout the ink was drawn against is never unrecoverable.
  8. Reflow strategy is phased.
    • MVP (Phase 2): absolute document coordinates + layout fingerprint (content_width). Honest baseline: ink is correct as long as layout width matches; the auto-snapshot from decision 7 makes that the common case for saved pages. If current width ≠ stored width, still render (best effort).
    • Phase 3: text-anchored clusters. New content-script command anchor-at-point {docX, docY} returns the nearest text position (raw offset + prefix/suffix, reusing the capture-position machinery in WebContentScript.swift) plus that anchor's current document rect. Each cluster stores its anchor + the rect at capture time. On init, relayout (the existing ResizeObserver/fonts/MutationObserver hooks), and width change, Swift batch-re-resolves all anchors and translates each cluster by (newRect − oldRect). Vertical drift — the dominant reflow effect — is fully corrected; horizontal is usually stable within a text column. This is exactly how highlights already survive reflow, applied at cluster granularity. Unresolvable anchors fall back to stored coordinates.
  9. Ink tool core is extracted, not duplicated. InkController_iOS splits:
    • InkToolState (shared, @Observable): tool/color/width-slot state, InkWidthSettings UserDefaults blob, pkTool(widthScale:) construction, eraser mode, finger-drawing toggle, pencil double-tap preference handling.
    • A small InkPaletteHost protocol (isActive, undo/redo/canUndo/canRedo, clearCurrentPage, done) adopted by both the PDF controller and the new WebInkController_iOS, so InkToolPalette_iOS renders either untouched.
    • PDF behavior must be byte-for-byte identical after the refactor (Tests/InkPersistenceTests.swift + manual regression pass gate Phase 1).
  10. Ink mode is modal, like PDF. While active the canvas intercepts all touches: links, text selection, and note interactions are inert (matches both the PDF implementation and GoodNotes). Entering ink mode dismisses open popovers and auto-collapses the sidebar (reuse the isActive didSet behavior, InkController_iOS.swift:90-117). Two-finger-tap undo comes free from PencilKit. UIPencilInteraction attaches to the web container view (mirroring the always-mounted-PDFView rationale, PdfKitView_iOS.swift:51-54) and routes through the same PencilDoubleTapAction preference.
  11. Canvas pinned overrideUserInterfaceStyle = .light, same as PDF (InkOverlayProvider_iOS.swift:135): PencilKit dynamically inverts some ink colors in dark mode, but a web page's own theme is independent of the system theme — pinning light keeps ink WYSIWYG against whatever the page renders.

4Known, accepted limitations

Document in-app expectations; don't fight them.

Fixed/sticky elements: ink lives in document scroll space — strokes over position: fixed/sticky headers separate from them on scroll; strokes over internally-scrolling iframes/subregions don't track that inner scroll. (GoodNotes never faces this; no web-ink product solves it without freezing the page.)
Changed live content: a live page whose content changed since inking (news homepage) can only be best-effort re-anchored; the saved snapshot (decision 7) preserves ground truth.
Mid-pinch rendering: ink is bilinear-scaled until the gesture settles (PDF behaves the same; Notes too).

5Phases

Phase 0 — Spike

½ day · throwaway branch · validates the 3 load-bearing assumptions

  • PKCanvasView with a very large contentSize (~300k pt) — confirm memory stays flat while scrolling/drawing (PencilKit tile behavior).
  • KVO on WKWebView.scrollView.contentOffset + programmatic offset writes — confirm WebKit doesn't fight it and JS scroll events fire on programmatic sets.
  • scrollView.zoomScale KVO during native pinch — confirm it tracks VisualViewport's visualScale 1:1 on iPadOS 26 (they should be the same quantity; the plan's transform math assumes it).
Abort criteria: if (1) fails → fall back to banded canvases (design decision 1 revisited); if (2) fails → sync via the existing JS scroll messages instead (higher latency, still workable).

Phase 1 — Ink core refactor

~1 day · PDF must not change behavior

  • New Vellum/Platform/iOS/InkToolState.swift: extract tool/color/width/eraser/finger-toggle/double-tap state + pkTool(widthScale:) from InkController_iOS.swift (lines 9-83, 310-332 territory).
  • New InkPaletteHost protocol; InkToolPalette_iOS.swift generalized to it; PdfChrome_iOS toggle unchanged.
  • InkController_iOS adopts both, delegating state to InkToolState. No persistence-path changes.
  • InkRegistry_iOS (in PaneView_iOS.swift:16-28) keyed to hold either controller kind (the palette host protocol).
Gate: build + InkPersistenceTests green + manual PDF ink regression on simulator (draw/erase/undo/zoom/persist/restart).

Phase 2 — Web ink MVP

~3-4 days · the bulk: overlay, sync, persistence, chrome

  • New Vellum/Platform/iOS/WebInkController_iOS.swift (@Observable, InkPaletteHost): owns the canvas, activation lifecycle, debounced persist, flush hooks.
  • New Vellum/Platform/iOS/WebInkOverlay_iOS.swift: the PKCanvasView subclass + container; scroll sync both directions (decision 2); contentSize tracking via KVO on webView.scrollView.contentSize (native — no JS needed for doc height); pageZoom rescale (decision 3); native-pinch transform tracking (decision 4); light-mode pin.
  • WebViewRepresentable_iOS (in WebViewerView_iOS.swift): mount overlay above web view; attach UIPencilInteraction; expose controller to the SwiftUI layer.
  • Web chrome: ink toggle button (mirror PdfChrome_iOS.swift:136-140), palette overlay pinned bottom like PdfViewerView_iOS.swift:63-68; entering ink mode dismisses selection/note popovers.
  • New Vellum/Services/Web/WebInkStore.swift: WebInkRecord Codable (format in decision 5), WebInkIO actor, path resolution through WebStorageLayout (records dir, all three modes), atomic writes, per-path lock.
  • Load on document open (after handleInit); save debounced; flush on deactivate/tab-close/scene-background (hook where PDF's flushPendingInkAndWait is called); first stroke → promote saved.
  • Undo/redo/clear wired through canvas undoManager + drawingVersion bump (same observation trick, InkController_iOS.swift:178-205).
  • Tests: WebInkPersistenceTests.swift — record round-trip, zoom-1 normalization invariant (draw at zoom 1.5 → stored CSS px identical to zoom 1.0 equivalent), debounce/flush behavior, promote-to-saved.
Gate: simulator QA (finger drawing auto-enables in Simulator, as PDF does) on a long real article: draw → scroll (ink tracks) → zoom (crisp) → pinch (tracks) → kill app → relaunch → ink restored at correct positions.

Phase 3 — Reflow resilience (anchored clusters)

~2-3 days · JS + clustering + re-anchor

  • Content script (WebContentScript.swift): add anchor-at-point and batch resolve-anchors commands (reuse rangeFromRaw/resolveHighlight internals); add an ink-anchors-shifted report wired into the existing relayout (ResizeObserver / fonts.ready / MutationObserver, lines 2020-2079) and the 2000 ms hydration re-init.
  • Swift: spatial clustering of strokes at save time (proximity threshold, tuned ~48 CSS px); per-cluster anchor capture on stroke commit; re-anchor pass translating clusters on init/relayout/width-change; fall back to stored coordinates when an anchor doesn't resolve.
  • Rotation/split-screen: viewport width change triggers the same re-anchor pass once layout settles (piggyback on the existing settle machinery — the 400 ms / 1200 ms re-check pattern used by scroll restore).
  • Tests: clustering determinism; anchor round-trip against a fixture HTML page loaded in a test web view; translation math.
Gate: article inked in portrait stays attached to its paragraphs in landscape and in 50/50 split-screen; ink on a hydration-heavy page (e.g. a React docs page) lands correctly after reload.

Phase 4 — Ecosystem integration

~2 days

  • .vellumweb archives (WebArchive.swift): add optional ink.json entry + manifest sha256. Verify old readers skip unknown manifest entries before keeping formatVersion = 1; bump to 2 only if they hard-fail. Import merges by cluster id, newer updated_at wins (mirror mergeAnnotations, WebArchive.swift:677-698).
  • WebStorageMigrator (WebStorage.swift:452-625): move <key>.ink.json alongside records in local↔iCloud↔custom relocations; adoptRecordFile-style newer-wins merge for raced copies.
  • AI/region capture (captureRegionImage/capturePageImage, WebViewerView_iOS.swift:744-814): composite the ink via PKDrawing.image(from:scale:) over takeSnapshot output so what the model sees matches what the user sees.
  • Sidebar: "Handwriting" section for web docs (mirror InkPagesSection_iOS in PdfChrome_iOS.swift) — jump list of clusters, mapped to virtual pages via cluster anchor start_offset → page number; tap scrolls to cluster bounds.
  • Settings: nothing new expected — double-tap + finger-draw + width settings are already shared via InkToolState.

Phase 5 — Hardening + on-device QA

~1-2 days

  • Memory audit: very long pages (50k+ px), many clusters, repeated zoom-rescales (rescale allocates a new PKDrawing — watch peaks).
  • Deep-zoom crispness enhancement if warranted: super-sample bump (K = ceil of settled zoomScale, capped 4) after pinch settles — direct port of the PDF trick.
  • Edge sweep: pageshow back/forward cache restore; web-content-process crash → snapshot fallback path (loadSnapshotFallback) must reload ink too; offline snapshot vs live page must resolve to the same ink record (same normalized-URL key — verify the snapshot's proxy URL normalizes identically); YouTube-facade pages; find-in-page overlay under ink.
  • On-device checklist (needs a real Pencil — Simulator can't verify these): latency/ProMotion feel, pressure/tilt response, barrel double-tap, palm rejection while resting hand, scribble non-interference, split-screen + Stage Manager resize while ink mode is active.

7Open questions (decide during Phase 0/2, defaults chosen)

  1. Should toolbar zoom stay enabled while ink mode is active? Default: yes (rescale is cheap and discrete). Native pinch while actively inking: default disabled (pencil draws, fingers pan — matching PDF, where PDFView pinch still works but ink canvases only pan); revisit if it feels limiting.
  2. Cluster granularity for the sidebar jump list — per cluster vs per virtual page. Default: per cluster, grouped by page header.
  3. Whether macOS should ever render web ink (read-only). Out of scope here; the sidecar format is deliberately platform-neutral so it stays possible.
